# Editorial Guide

This guide keeps BS Engineering documents from becoming dry, generic or overly corporate.

## Voice

The voice should be:

- professional, but human,
- clear, but not cold,
- confident, but not inflated,
- technical, but understandable,
- honest about limits,
- focused on real work and real learning.

## Avoid

Avoid phrases that sound like generic marketing copy:

- passionate professional,
- proven track record,
- highly motivated individual,
- cutting-edge solutions,
- world-class expertise,
- dynamic environment,
- strong problem-solving skills without examples.

## Prefer

Prefer concrete, grounded language:

- I worked with...
- This project explores...
- The problem is...
- The goal is...
- The current limitation is...
- This helps me understand...
- The next step is...

## Tone test

Before publishing a document, ask:

1. Does this sound like a real person?
2. Can I explain every claim in an interview?
3. Is there a clear reason why this document exists?
4. Does it show thinking, not only keywords?
5. Would a technical reader trust it?

## Best sentence pattern

A strong BS Engineering sentence usually connects three things:

```text
real context + technical action + practical purpose
```

Example:

> This project explores how production, warehouse, quality and calibration data could be connected in one lightweight manufacturing workflow.

That is better than:

> This project demonstrates advanced enterprise transformation capabilities.

The first sentence is specific. The second sounds impressive but empty.
